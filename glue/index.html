FinX Glue

Build on the orchestration layer.

Glue is the BIAN-aligned orchestration layer at the center of the FinX platform. It exposes a coherent canonical model over heterogeneous cores and ecosystem partners, so you can ship customer journeys without re-modeling the bank for every change.

Quickstart → API Reference

Get started in three steps

From zero to a working API call in roughly fifteen minutes. All quickstart examples target the sandbox environment with synthetic data.

01
Auth & tenant headers
Configure your X-Tenant-Id and bearer token. OAuth2 client credentials flow.
02
Environments
Sandbox, staging, production. Endpoint formats, rate limits, and data residency.
03
First API call
Initiate a party, retrieve it, and observe the canonical response shape.

Concepts

Glue is opinionated about three things: that the canonical model should follow BIAN, that extension should not require forking, and that orchestration should be observable end to end.

CONCEPT
BIAN Service Domains
How Glue maps endpoints to BIAN service operations and why that matters for coexistence.
CONCEPT
Canonical model
A single shape for parties, accounts, and arrangements across cores and connectors.
CONCEPT
Extensibility model
Add tenant-specific attributes without breaking schema contracts or upgrade paths.

API surfaces

Six surface areas, each grouped by BIAN service domain. Counts indicate operations exposed in v1.

Party Lifecycle Management
Initiate, update, and close parties. Identity verification and AML hooks.
14 ops
Account Management
Open, configure, and close current and savings accounts on the configured core.
11 ops
Payments
Initiate, query, and reverse payments across schemes and agent banks.
9 ops
Document Directory
Upload, retrieve, and reference customer documents stored in the document layer.
6 ops
IDV connectors
Initiate identity verification through Jumio, Thirdstream, and others.
5 ops
AML connectors
Sanctions, PEP, and adverse media screening via ComplyAdvantage.
4 ops
