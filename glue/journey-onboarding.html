Customer onboarding and KYC journey

6 API calls ~3 min PLM · IDV · AML

This journey takes a prospect from first contact through identity verification and AML screening to an active party on your bank's books. It composes six Glue endpoints across three BIAN service domains and two ecosystem partners. Most banks implement this exact sequence on day one.

Overview

Onboarding is the canonical FinX journey: it touches Party Lifecycle Management on the bank side, Document Directory for evidence storage, Jumio for identity verification, and ComplyAdvantage for sanctions and PEP screening. Glue orchestrates these as a single linear flow, polls the asynchronous IDV result, and lifts the party to active status only when both checks complete cleanly.

API sequence

Customer onboarding · 6 calls
PLM → DOCS → IDV → AML → PLM
1
POST /v1/parties/initiate PLM · BIAN
Create a draft party from the prospect's submitted attributes. Returns cust_8a2f with status draft.
2
POST /v1/documents Document Directory
Upload the customer's ID document. Glue persists the binary in the document layer and returns a reference token to attach to subsequent calls.
3
POST /v1/idv/initiate Jumio
Initiate identity verification with the document reference and party ID. Returns an idv_session handle.
4
GET /v1/idv/{id}/result Poll until complete
Poll every 2s with exponential back-off, or subscribe to the idv.completed webhook. Most sessions resolve in 8–20 seconds.
5
POST /v1/aml/screen ComplyAdvantage
Screen the verified identity against sanctions, PEP, and adverse-media lists. Returns a synchronous decision plus a list of any matches.
6
PATCH /v1/parties/{id}/lifecycle Update to active
Lift the party from draft to active. The same endpoint is used later for suspension, reactivation, and closure.
Polling guidance The IDV result endpoint is asynchronous. Begin polling no sooner than 2 seconds after initiation, then back off as 2s, 4s, 8s, 16s capped at 30s. Always honour the Retry-After header when present. For volumes above 1k onboardings/day, prefer the idv.completed webhook.

Example: end-to-end cURL

The following sequence creates a party, uploads a document, and starts an IDV session. The remaining three calls follow the same pattern.

cURL
# 1. Initiate party
curl -X POST https://api.ustfinx.com/v1/parties/initiate \
  -H "Authorization: Bearer $TOKEN" \
  -H "X-Tenant-Id: bank_a" \
  -H "Content-Type: application/json" \
  -d '{
    "givenName": "Asha",
    "familyName": "Mehta",
    "dateOfBirth": "1992-04-12",
    "residentialAddress": { "country": "AE", "city": "Abu Dhabi" }
  }'

# → { "id": "cust_8a2f", "status": "draft" }

# 2. Upload ID document
curl -X POST https://api.ustfinx.com/v1/documents \
  -H "Authorization: Bearer $TOKEN" \
  -F "partyId=cust_8a2f" \
  -F "type=passport" \
  -F "file=@./passport.jpg"

# 3. Initiate IDV
curl -X POST https://api.ustfinx.com/v1/idv/initiate \
  -H "Authorization: Bearer $TOKEN" \
  -d '{ "partyId": "cust_8a2f", "documentRef": "doc_3df1" }'
Idempotency Every POST in this sequence accepts an Idempotency-Key header. Reuse the same key on retries to avoid duplicating parties, documents, or IDV sessions. Keys are honoured for 24 hours.
